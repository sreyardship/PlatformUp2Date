# The ssh-os-release source stays MINA-in-native, guarded by a native integration test and committed reachability metadata

The `ssh-os-release` current source ([ADR-0016](0016-ssh-os-release-current-source.md))
uses Apache MINA SSHD as an in-process SSH client. The backend ships as a GraalVM
native image, and MINA relies on heavy reflection that GraalVM cannot see at build
time. In the native image MINA's BouncyCastle named-provider path reflectively calls
the 2-arg `java.security.KeyPairGenerator.getInstance(algorithm, provider)` during
`SshClient.setUpDefaultClient()`; that method is not registered, so it throws
`NoSuchMethodException`. Worse, the client was built in the source's *constructor*,
which runs at `VersionSourceResolver` build time — so the failure escaped the per-app
isolation [ADR-0016](0016-ssh-os-release-current-source.md) promised and crashed the
whole fleet scrape (and NPE'd the MCP `get_application`). This shipped because the only
SSH test was a plain JVM JUnit test, which the native `quarkusIntTest` stage never ran.

We decided to **keep MINA in the native image** rather than retreat to a JVM image or
shell out to the system `ssh` binary, and to make it work via **committed GraalVM
reachability metadata generated by the tracing agent**, locked in by a **native
`@QuarkusIntegrationTest`** that exercises a real SSH handshake against an embedded
ed25519 server. The construction of the `SshClient` moves out of the constructor into
`version()` (per-call, try-with-resources), so every SSH/environment failure surfaces
per-app and is counted FAILED — restoring the isolation contract — while genuine
config errors (blank host/user) still fail the deploy fast at the factory.

## Considered Options

- **Shell out to the system `ssh` binary** — rejected for an enterprise posture. It
  removes the Java SSH stack and all native-reflection metadata, but introduces a
  process-execution surface, makes the artifact non-self-contained, couples us to the
  image's PATH and openssh version, and is incompatible with a minimal/distroless base
  image (it would *add* a shell + ssh client and their CVEs). A review board is more
  likely to object to "the JVM forks `ssh`" than to a vetted in-process library.
- **Run this backend on the JVM image instead of native** — rejected: native is a hard
  requirement here. It is the lowest-brittleness option (MINA just works on the JVM, no
  metadata) and remains the escape hatch if the metadata maintenance ever outweighs the
  native benefits.
- **Hand-written `reflect-config.json`** — rejected: most brittle and highest effort,
  chasing one `NoSuchMethodException` per native build.
- **Disable MINA's BouncyCastle registrar** (`org.apache.sshd.registerBouncyCastle=false`)
  — *attempted and rejected*. The premise was that JDK 21+/25 has native EdDSA, so ed25519
  would work without BouncyCastle. It does not hold for MINA SSHD 2.17: MINA has **no
  JDK-native EdDSA backend** — its EdDSA support comes only from BouncyCastle or the optional
  net.i2p library. With the BC registrar disabled, `SecurityUtils.isEDDSACurveSupported()`
  returns `false` and loading an OpenSSH ed25519 key throws
  `NoSuchAlgorithmException: Unsupported key type (ssh-ed25519)`. Since the production host
  keys are ed25519, BouncyCastle is required; disabling it would break the source. The
  BC-inclusive committed metadata is therefore retained as the accepted outcome.

## Consequences

- A MINA SSHD upgrade can invalidate the committed reachability metadata. This is a
  bounded, visible chore rather than a silent prod failure: the native `quarkusIntTest`
  stage exercises the SSH path in the native binary, so stale metadata fails the build.
  Regenerate by re-running the tracing agent against the SSH integration test.
- SSH test coverage must use **ed25519** (the production host-key type per
  [ADR-0016](0016-ssh-os-release-current-source.md)), not just RSA, or the trace and the
  guard miss the algorithm path that actually runs in production.
- The `SshClient` is built per `version()` call. Scrapes are infrequent (bounded by the
  scrape budget), so the cost is negligible and buys full per-app failure isolation and
  no shared mutable client state across replicas.
- BouncyCastle (`bcpkix`) stays a required dependency: MINA SSHD 2.17 needs it for ed25519
  (the production host-key type), so it cannot be dropped (see the rejected disable-BC option
  above). It is registered at native build time (`-H:AdditionalSecurityProviders`) and added
  to the runtime provider list by `BouncyCastleProviderInstaller` (native only).
- MINA SSHD is initialised at image **run time** (`--initialize-at-run-time=org.apache.sshd`
  via `quarkus.native.additional-build-args`). Its class initializers otherwise create
  disallowed image-heap objects (a global `JceRandom` `SecureRandom` in a static field) and
  reference an i2p-eddsa decoder backend we do not bundle. Deferring the whole package to
  runtime is robust against MINA upgrades and avoids per-class whack-a-mole; ed25519 still
  flows through the JDK/BouncyCastle providers.

## Regenerating the reachability metadata

The committed config lives at
`backend/src/main/resources/META-INF/native-image/org.yardship/platform-up-2-date/reachability-metadata.json`.
Regenerate it after a MINA SSHD upgrade (or when the native `quarkusIntTest` stage fails
with a reflection/`NoSuchMethodException`):

1. Run the in-process SSH integration test under the GraalVM tracing agent. The agent must
   run on the GraalVM JDK; an access filter (`backend/native-agent-access-filter.json`)
   keeps only the SSH client + crypto path and drops the test harness, the embedded MINA
   *server*, JUnit, Gradle and WireMock:

   ```bash
   cd backend
   gradle test --tests '*SshOsReleaseCurrentSourceIT' \
     -PnativeAgentDir="$PWD/build/native-agent-config" \
     -PnativeAgentFilter="$PWD/native-agent-access-filter.json" \
     --rerun-tasks
   ```

2. Copy the result into place:

   ```bash
   cp build/native-agent-config/reachability-metadata.json \
     src/main/resources/META-INF/native-image/org.yardship/platform-up-2-date/
   ```

3. Rebuild native and run the guard (`ci/bin/build-native.sh`, or the equivalent
   `gradle ... -Dquarkus.native.enabled=true build quarkusIntTest`). Iterate on any
   remaining `--initialize-at-run-time` needs (see `quarkus.native.additional-build-args`)
   until `SshOsReleaseResourceIT` is green in native.
